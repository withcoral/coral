---
title: "Sources and workspaces"
description: "Understand Coral's core resource model: sources, source specs, workspaces, and SQL tables."
---

Coral's user-facing model is deliberately small. These are the terms to keep straight.

## Source

A source is an external system or local dataset that Coral can query.

Examples:

- GitHub
- Slack
- Stripe
- local JSONL files
- local Parquet files

In public docs, Coral uses the term `source` consistently. Avoid reading extra meaning into implementation terms such as `connector` or `provider`.

## Source spec

A source spec is the configuration Coral validates and installs for one source.

A source spec declares things like:

- the source name and version
- the backend type, such as `http`, `jsonl`, or `parquet`
- tables exposed by that source
- required filters, variables, and secrets
- request or file-location details

Bundled sources ship with source specs in the repo. Coral can also import custom source specs from YAML.

## Workspace

A workspace is the local set of installed sources Coral queries together.

Today the default experience is a single local workspace. Installed sources, variables, and secrets are all scoped to that local workspace state.

## Table

Coral exposes source data as SQL tables.

Each source owns a schema, and each declared table appears inside that schema.

Examples:

- `github.releases`
- `slack.channels`
- `otel.logs`

The system catalog tables under the `coral` schema help you discover those tables and columns.
