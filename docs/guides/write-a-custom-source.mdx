---
title: "Write a custom source"
description: "Write a source spec to connect any HTTP API or local dataset to Coral, then query it through SQL."
---

Coral ships with [bundled sources](/reference/bundled-sources) for common systems, but you are not limited to them. If the system you need is not bundled, you can write a source spec in YAML and import it.

A source spec tells Coral how to connect to an API or read a local dataset, what tables to expose, and what columns each table has. Once imported, the source works exactly like a bundled one — same SQL, same system catalog, same MCP surface.

## Agent-driven authoring

Source authoring works well as an agent-driven workflow. Point your coding agent at the Coral CLI and the [source spec reference](/reference/source-spec-reference), give it the API docs or describe the endpoints you want, and let it iterate:

1. The agent writes or updates the source spec YAML
2. Imports it with `coral source import ./my-source.yaml`
3. Validates with `coral source test my_source`
4. Inspects `coral.tables` and `coral.columns` to check the shape
5. Refines and repeats until the tables look right

The agent needs access to the CLI and the spec reference — no MCP connection is required for authoring.

## Example: local JSONL source

The fastest way to see a custom source working is with a local file.

### 1. Create sample data

```shellscript
mkdir -p demo-data
cat > demo-data/messages.jsonl <<'EOF'
{"type":"user","sessionId":"s1","text":"hello"}
{"type":"assistant","sessionId":"s1","text":"world"}
EOF
```

### 2. Write the source spec

Create `local-messages.yaml`:

```yaml
name: local_messages
version: 0.1.0
dsl_version: 3
backend: jsonl
tables:
  - name: messages
    description: Demo messages
    source:
      location: file:///absolute/path/to/demo-data/
      glob: "**/*.jsonl"
    columns:
      - name: type
        type: Utf8
      - name: sessionId
        type: Utf8
      - name: text
        type: Utf8
```

Replace `/absolute/path/to/demo-data/` with the real absolute path on your machine.

### 3. Import and validate

```shellscript
coral source import ./local-messages.yaml
coral source test local_messages
```

### 4. Query it

```shellscript
coral sql "
  SELECT type, sessionId, text
  FROM local_messages.messages
  ORDER BY sessionId, type
"
```

## Example: HTTP API source

To connect an HTTP API, the source spec declares the base URL, auth, endpoints, and response shape.

```yaml
name: demo_api
version: 0.1.0
dsl_version: 3
backend: http
base_url: "{{variable.API_BASE|https://api.example.com}}"
auth:
  headers:
    - name: Authorization
      from: template
      template: Bearer {{secret.API_TOKEN}}
tables:
  - name: messages
    description: Messages from the API
    request:
      method: GET
      path: /messages
    response:
      rows_path:
        - data
    pagination:
      mode: link_header
      page_size:
        default: 50
        max: 100
        query_param: per_page
    columns:
      - name: id
        type: Utf8
      - name: text
        type: Utf8
      - name: created_at
        type: Utf8
```

When you run `coral source import`, Coral prompts for the variable and secret values interactively.

For the full set of HTTP response, pagination, auth, and column fields, see the [source spec reference](/reference/source-spec-reference).

## Tips

- **Start small.** Begin with one table and a few columns. Validate with `coral source test`, check `coral.columns`, then expand.
- **Use the system catalog.** `coral.tables` and `coral.columns` show you exactly what Coral sees after an import — including required filters and column metadata. See [System catalog](/concepts/system-catalog).
- **Look at bundled sources for patterns.** The bundled source specs in the repo under `sources/` are working examples of HTTP pagination, response parsing, auth, and variables.
